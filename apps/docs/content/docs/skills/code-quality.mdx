# Code Quality

Enforce consistent code style and catch issues early through automated linting, formatting, and structured code review practices.

## Linting

Use ESLint with TypeScript support to catch type errors, unused imports, and common bugs.

Key rules to enforce:

- **No unused variables or imports** — Keeps the codebase clean
- **Explicit return types on exported functions** — Improves API readability
- **No console.log in production code** — Use a proper logger
- **Consistent import ordering** — Group by external, internal, relative

## Formatting

Use Prettier with a shared configuration across all packages in the monorepo.

Configuration highlights:

- Single quotes
- 2-space indentation
- 100 character line width
- Trailing commas (ES5 compatible)
- Semicolons enabled

Run formatting before every commit:

```bash
npm run format
```

## Code Review Checklist

- [ ] All new code has corresponding tests
- [ ] No TypeScript errors (`npm run typecheck` passes)
- [ ] No lint errors (`npm run lint` passes)
- [ ] Code is formatted (`npm run format:check` passes)
- [ ] Complex logic has inline comments explaining the "why"
- [ ] No hardcoded values — use constants or config
- [ ] Error handling covers edge cases

## Anti-Patterns

- **Ignoring lint errors** — Fix them or explicitly disable with a reason
- **Inconsistent formatting** — Let Prettier handle it, don't manually format
- **Large PRs without context** — Break into smaller, focused changes
- **Missing type definitions** — Avoid `any`, use `unknown` when necessary
